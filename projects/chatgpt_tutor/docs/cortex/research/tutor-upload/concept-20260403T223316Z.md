## Summary

The fastest path is a single Node.js Express + Multer service in Docker, storing images to local disk with a cron cleanup. The session pack page serves images as `<img>` tags that the parent can **drag directly into ChatGPT's message bar** — this works natively on all browsers for Pro subscribers with zero extra steps. Clipboard copy (navigator.clipboard.write) is a nice secondary option but drag-and-drop is simpler and requires no JavaScript interaction. The infrastructure is straightforward: add a service to docker-compose.yml, add a Caddyfile block, create the Cloudflare DNS record.

## Findings

### 1. Image Transfer UX — Drag-and-Drop Wins

**Drag-and-drop from browser to ChatGPT works natively.** The parent opens the session pack page in one browser tab and ChatGPT in another, then drags each image from the pack page directly into ChatGPT's message bar. This is confirmed working on Chrome, Edge, Safari, and Firefox for paid ChatGPT accounts.

**Clipboard copy (click-to-copy) is the backup.** `navigator.clipboard.write()` with `ClipboardItem` supports copying PNG blobs to clipboard across all major browsers since March 2025. The parent clicks "Copy" on an image, then Ctrl+V into ChatGPT. Works but requires two actions per image vs. one drag.

**Recommendation:** Primary UX is drag-and-drop. Add a "Copy to clipboard" button per image as secondary. No download needed.

### 2. Infrastructure Pattern — Docker Compose Service

### 3. Storage — Local Filesystem Over Garage S3

Garage S3 is available (port 3900) but it's **not on the agent-stack network**, so containers can't reach it via DNS. Adding it to the network is possible but unnecessary complexity for this use case.

**Local filesystem in a Docker volume** is simpler:
- Mount a named volume (`tutor_upload_data:/app/uploads`)
- Images stored as files with UUID names
- Cron job or in-process timer deletes files older than 24 hours
- No bucket creation, no S3 client library, no credentials

### 4. Tech Stack — Node.js + Express + Multer

Node 20 is available on the server. The minimal stack:
- **Express** — HTTP server, static file serving
- **Multer** — multipart file upload handling
- **nanoid** — session pack ID generation (short, URL-safe)
- **No database** — session packs are directories on disk. A pack is a directory named by its ID containing the uploaded images + a manifest.json with metadata.

**Alternative considered:** Python + Flask. Rejected because the existing showcase service uses Node.js, so this follows the established pattern.

### 5. DNS — Cloudflare A Record

Create an A record for `tutorupload.calenwalshe.com` pointing to the server IP, proxied through Cloudflare:
```bash
curl -X POST "https://api.cloudflare.com/client/v4/zones/$CF_ZONE_ID/dns_records" \
  -H "Authorization: Bearer $CF_API_TOKEN" \
  -H "Content-Type: application/json" \
  --data '{
    "type": "A",
    "name": "tutorupload",
    "content": "144.202.81.218",
    "proxied": true,
    "ttl": 1
  }'
```

### 6. Session Pack Page UX

The pack page at `tutorupload.calenwalshe.com/pack/<id>` shows:
- All images in the pack as large, draggable `<img>` elements
- Each image has a "Copy to clipboard" button
- Images are numbered/labeled (Problem 1, Problem 2, etc.)
- A "Download all" zip option as fallback
- Clean, minimal layout — no distractions

**Key detail:** Images must be served with proper CORS and content-type headers so browsers allow drag-and-drop to external sites and clipboard API access.

## Trade-offs

| Decision | Upside | Downside |
|----------|--------|----------|
| Local filesystem over S3 | Zero config, no library deps, volume-backed | Not distributed, but single-user so doesn't matter |
| Node.js over Python | Matches existing agent-showcase pattern | Python would work equally well |
| Drag-and-drop as primary UX | One action per image, zero JS needed on user side | Requires side-by-side browser tabs |
| 24-hour auto-expiry | Keeps disk clean, no manual management | If parent wants to reuse a pack next day, need to re-upload |
| No auth | Zero friction for the parent | Anyone with the URL can view — mitigated by random pack IDs |

## Recommendations

1. **Use nanoid for pack IDs** — 12-char URL-safe random strings. Effectively unguessable for a single-user app.
2. **24-hour expiry** with a simple setInterval cleanup in the Node process. No cron needed.
3. **Single-page upload UI** — drop zone + file picker, creates a pack immediately, redirects to the pack page.
4. **Pack page optimized for drag-and-drop** — large images, numbered, minimal chrome.
5. **No database** — pack = directory on disk. Manifest is a JSON file in the directory.
6. **Camera capture on mobile** — use `<input type="file" accept="image/*" capture="environment">` to enable camera on phones. Free with HTML — no extra code.
