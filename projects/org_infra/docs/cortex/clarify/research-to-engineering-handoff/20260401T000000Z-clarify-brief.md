## Goal

Build a multi-agent intelligence system that takes a product problem as input and produces a complete, validated context package — covering research, product pitch, evaluation criteria, and test harness concept — that a downstream engineering system (MetaGPT or equivalent) can execute against without needing to do any discovery work itself.

---

## What This System Does

The system owns everything **before** implementation. Its output is a structured handoff package consumed by an engineering system. It does not care which engineering system receives it — MetaGPT, a human engineering team, or something else.

**Agent roles in scope:**
- **Research Orchestrator** — coordinates all agents, gates phase transitions, assembles final handoff package
- **UX Research agent** — user signal synthesis, persona generation, problem validation
- **PM agent** — product pitch, requirements definition, prioritization, scope; direct stakeholder communication (drafts and sends comms, manages async review cycles, collects responses — no human intermediary for routine feedback loops)
- **Data Science agent** — quantitative feasibility, data availability, experiment design
- **Evaluation agent** — success criteria, test harness concept, what "done" looks like
- **Pressure Testing agent** — adversarial review of assumptions, red-teaming, stress-testing the pitch
- **Feedback Synthesis agent** — collects, structures, and integrates external stakeholder feedback on the vision and pitch; surfaces conflicts between internal findings and external input

**The handoff package (system output):**
- Validated problem statement + user research summary
- Product pitch / brief
- Prioritized requirements (what to build, what not to)
- Evaluation criteria (measurable definition of success)
- Test harness concept (structure and intent of tests, not implementation)
- Risk log + open assumptions for engineering to resolve
- Feedback synthesis report — what external stakeholders said, where it aligns with internal findings, where it conflicts

---

## Non-Goals

- Does not write implementation code
- Does not own production engineering, DevOps, or deployment
- Does not implement the test suite — only designs its structure and intent
- Does not manage the engineering system that receives the handoff
- Does not do visual design / wireframing (open question — see below)
- Does not operate post-handoff — once the package is delivered, this system's job is done
- Does not replace the engineering system — assumes one exists downstream

---

## Constraints

- System must produce a handoff package in a format a downstream engineering system can consume without human translation
- Stops at the production boundary — all implementation is engineering's responsibility
- Built on Meta's existing infra — not a standalone external system
- Downstream engineering system is treated as a black box — the handoff format is the API contract
- Must be cross-product: not tied to one team or product area

---

## Assumptions

- A downstream engineering system exists (MetaGPT-like or equivalent) that can consume the handoff package
- "Pressure testing" means adversarial validation of assumptions — red-teaming the pitch, stress-testing personas, challenging the problem statement — not load testing or performance testing
- The system is agentic (multi-agent orchestration), not a manual workflow or form-based process
- Research Orchestrator is the gate agent — every phase transition requires its approval
- The handoff package is a structured artifact, not a freeform document
- PM agent communicates directly with stakeholders (Model B) — no human intermediary for routine feedback cycles; human gates major decisions only
- Requires a stakeholder-facing interface (Slack, email, or async review platform) that the PM agent can write to and read from

## Open Questions

- **Who provides the feedback?** Leadership stakeholders, cross-functional partners (eng, design, policy), external users, domain experts — or some defined combination? This determines how the Feedback Synthesis agent collects input (async written responses, structured interviews, review sessions).
- **When in the process does feedback happen?** After the product pitch is drafted? After the full package is assembled but before handoff? Multiple times? The placement determines whether feedback can reshape the research or just annotate it.
- **Does feedback block handoff or annotate it?** If stakeholders flag a conflict with the pitch, does the system need to resolve it before handing off, or does it pass the conflict to engineering as a known open item?
- **Does visual design (wireframes, UX flows) belong in scope?** The current framing excludes implementation but includes the "concept for a harness around engineering." Does that extend to visual/interaction design, or does design get handed to engineering as a specification without mockups?
- **What is the handoff format?** Structured JSON schema + prose documents? A Git repo with typed artifacts? A specific internal Meta format? This determines how the downstream engineering system reads the package.
- **What triggers a handoff?** Is the handoff triggered by a quality gate (Orchestrator declares the package complete), a human decision, a time box, or something else?
- **How does the system receive new work?** What's the intake mechanism — a Slack message, a form, an API call, a meeting summary?
- **What does the Pressure Testing agent specifically challenge?** The product pitch? The user research findings? The technical feasibility? All of the above at a defined phase?
