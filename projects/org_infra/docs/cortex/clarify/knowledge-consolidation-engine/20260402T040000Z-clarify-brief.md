## Idea (verbatim)

> Build the Consolidator — the async agent that runs between sessions to promote claims through maturity levels (L0 raw → L1 extracted → L2 validated → L3 actionable), detect contradictions, decay stale facts, and compress knowledge. This is the missing piece in the autonomous R&D platform — without it, the knowledge store accumulates raw observations forever and never graduates them into actionable intelligence. Research needed on: how Letta, Bedrock, Cognee, and Mem0 actually implement their consolidation pipelines technically, what the right promotion heuristics are, whether LLM involvement is required or rule-based is sufficient, how to handle staleness decay across domains, and where lossy compression is safe vs. destructive.

---

## Goal

A standalone Consolidator component that reads L0/L1/L2 claims from the knowledge store, promotes worthy claims upward, detects and flags contradictions, decays stale facts, compresses low-value knowledge, and regenerates Markdown projections — running as a scheduled background process without coupling to any other component.

---

## What This System Does

The Consolidator is one component in the composable autonomous R&D platform. It implements the "sleep-time compute" pattern: between sessions, it refines the knowledge store's contents so the next session starts with higher-quality, more organized knowledge.

**Operations:**
- **Extract** (L0 → L1): Parse raw observations into atomic facts with provenance
- **Validate** (L1 → L2): Corroborate extracted facts against independent sources
- **Promote** (L2 → L3): Elevate validated knowledge to actionable status
- **Detect contradictions**: Find claim pairs that conflict, flag for resolution
- **Decay stale facts**: Demote or expire claims whose sources have changed or gone missing
- **Compress**: Merge redundant claims, summarize low-value clusters
- **Trigger projection regeneration**: After consolidation, regenerate Markdown views

**Interface (store API only):**
```
READS:  Claim (all levels), Evidence, Contradiction
WRITES: Claim (promoted/demoted/compressed), Contradiction (new), Artifact (projections)
```

## Non-Goals

- Does not discover problems (that's the Discoverer component)
- Does not execute research or implementation (that's Executors)
- Does not evaluate execution results (that's the Evaluator)
- Does not ingest new data from external sources (that's Perceivers)
- Does not make decisions about what to build (that's the Loop Controller)
- Does not replace human judgment on contested contradictions — flags them for review

---

## Constraints

- Must be a standalone script: `python -m src.consolidator --db knowledge.db`
- Must talk to the store API only — no coupling to perceivers, projectors, or executors
- Must be independently testable with synthetic claims (mock store)
- Budget target: <$0.50 per consolidation cycle at 1000 claims
- Must work with SQLite store now and Postgres later (uses StoreProtocol, not implementation)
- Must not silently overwrite or delete claims — all changes are promotions, demotions, or flagged contradictions
- Must produce an audit trail: every promotion/demotion/contradiction has a reason

---

## Assumptions

- The six protocols (Claim, Evidence, Decision, Experiment, Problem, Artifact) are stable and won't change during Consolidator development
- The SQLite store adapter is correct and tested (40 tests passing)
- LLM calls (Claude Sonnet/Haiku or Gemini) are available for consolidation if needed
- The perceiver will have populated the store with enough L0 claims to test against
- Promotion thresholds will need calibration from real data — initial values are heuristics
- Contradiction detection does not need to be perfect — false positives are preferable to missed contradictions (human reviews flagged conflicts)

## Open Questions

1. **What algorithm detects contradictions without O(n²) pairwise comparison?** Embedding clustering? Entailment classification? Topic-scoped comparison? Need to find what Letta/Bedrock/Cognee actually do technically.

2. **What counts as "independent corroboration" for L1 → L2 promotion?** Two claims from the same perceiver run are not independent. Two claims from different sources expressing the same fact in different words — how do you detect semantic equivalence without LLM? Is embedding cosine similarity sufficient?

3. **What is the minimum viable LLM involvement?** Can L0→L1 extraction be rule-based (NER, regex, pattern matching) with LLM only for ambiguous cases? What's the cost per cycle at 100/1000/10000 claims for full-LLM vs. hybrid vs. rule-only?

4. **What are the right staleness decay rates per domain?** Codebase claims decay fast (code changes daily). Competitive intel decays medium (weekly). User preferences decay slow (monthly). Are these the right ballparks? How do production systems handle domain-specific decay?

5. **Where is lossy compression safe vs. destructive?** Compressing "Feature Y caused 12% regression in segment Z on March 15" to "Feature Y caused regression" loses the actionable detail. What's the heuristic for "this claim must keep full fidelity" vs. "this can be summarized"?

6. **Should the Consolidator run synchronously (blocking) or as a background process?** Letta's sleep-time compute is async. Our `/schedule` supports cron. But what if consolidation takes 5 minutes and a perceiver writes new claims mid-cycle?

7. **How do you handle the cold-start problem?** With <50 claims, statistical corroboration is meaningless. What are the right promotion rules for a sparse store vs. a mature one?

## Next Research Steps

1. **Deep-dive Letta sleep-time compute implementation** — find the actual code/architecture, not just the concept paper. How does it decide what to reorganize? What LLM calls does it make? What's the latency and cost?

2. **Deep-dive Amazon Bedrock AgentCore extraction/consolidation pipeline** — the specific extraction prompts, consolidation logic, contradiction resolution mechanism, compression strategy. Get as close to implementation detail as possible.

3. **Survey entailment/contradiction detection approaches** — NLI models (e.g., DeBERTa-MNLI), embedding-based methods, LLM-as-judge. Compare cost, accuracy, and latency for claim-pair contradiction detection at 1000 claims.

4. **Research semantic deduplication/clustering for claims** — how to detect that two differently-worded claims express the same fact. Embedding cosine similarity thresholds? Sentence-BERT? Cheaper alternatives?

5. **Find or design staleness decay functions with empirical backing** — look for any production system that documents its decay rates by domain. If none exist, design a reasonable default and flag it as "needs calibration."

6. **Cost model: LLM-hybrid consolidation at 1000 claims** — estimate token counts for extraction, contradiction detection, promotion decisions. Compare Claude Haiku, Sonnet, Gemini Flash costs. Identify which operations need quality (Sonnet) vs. which can be cheap (Haiku/Flash).
