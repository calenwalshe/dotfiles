## Constraint: Nothing Proprietary

- Open source with inspectable code, OR

No black boxes. No "proprietary algorithms optimized for common use cases." If we can't read the source or write it ourselves, it doesn't go in.

## Summary

The Consolidator is a 7-phase hybrid pipeline. Expensive operations (contradiction detection, corroboration) use open-source NLI models and sentence-transformers — no LLM needed. LLM calls (Claude/Gemini) are reserved for promotion validation only (~15-30 calls/cycle). Everything is implementable from source we can read.

**Stack:**
- `sentence-transformers/all-MiniLM-L6-v2` — claim embeddings (Apache 2.0, local)
- `cross-encoder/nli-MiniLM2-L6-H768` — contradiction detection (Apache 2.0, local)
- `hdbscan` — clustering (BSD 3-Clause, local)
- Claude API (Haiku for bulk, Sonnet for judgment) — promotion validation only
- SQLite/Postgres — store (public domain / PostgreSQL License)

## Finding 1: Letta Sleep-Time Compute (Open Source, Apache 2.0)

**Source:** Letta GitHub repo — `letta/agents/voice_sleeptime_agent.py`, `letta/agents/letta_agent.py`

**What it actually is (from source code):**

A separate agent that shares memory blocks with the primary agent. It has three tools with explicit flow rules:

```python
# From voice_sleeptime_agent.py — the actual tool rules:
agent_state.tool_rules = [
    InitToolRule(tool_name="store_memories"),           # must start here
    ChildToolRule(tool_name="store_memories", children=["rethink_user_memory"]),
    ContinueToolRule(tool_name="rethink_user_memory"),  # can loop
    TerminalToolRule(tool_name="finish_rethinking_memory"),  # must end here
]
```

**The flow:** `store_memories` → `rethink_user_memory` (loops N times) → `finish_rethinking_memory`

**What `rethink_user_memory` does:** Rewrites a memory block. The LLM decides what to change based on conversation history and current block content. There are no explicit heuristics — it's pure LLM judgment via function calling.

**Trigger:** Every N conversation steps (configurable, default 5). The sleep-time agent processes new messages asynchronously.

**What we take from this:**
- The three-tool-with-flow-rules pattern is clean and stolen directly
- LLM-for-everything is Letta's choice because their memories are unstructured text blocks. Ours are typed Claims with provenance — we can be smarter
- The trigger mechanism (every N steps) maps to "run after each perceiver batch"

**What we don't take:**
- Full LLM dependence for all operations — too expensive for structured data
- Block-based memory model — we use typed Claims, not text blobs

## Finding 2: Mem0 Consolidation (Open Source, Apache 2.0)

**Source:** Perplexity research on Mem0 internals

**How Mem0 consolidates:**
1. Retrieve top similar memories (cosine > 0.85 via embeddings)
2. Merge via LLM prompt that resolves conflicts (e.g., "likes pizza" ≈ "loves pizza")
3. Periodic consolidation averages vectors and updates relevance scores (+0.1 per query match)
4. Automatic filtering prevents bloat — decay for irrelevant data
5. 12x token reduction via cached retrieval

**What we take:**
- Cosine 0.85 threshold for semantic equivalence (production-validated)
- Cosine 0.9 for near-duplicate deduplication
- Relevance scoring based on query frequency

**What we don't take:**
- LLM for merging — our Claims have structured fields, we can merge programmatically

## Finding 3: Cognee (Open Source, Apache 2.0)

**Source:** Perplexity research on Cognee architecture

**What Cognee does:**
- Embeds and chunks ingested data
- Extracts facts automatically during ingestion
- Resolves contradictions
- Splits facts into static (long-term) and dynamic (recent context)

**What we take:**
- Static vs. dynamic split maps to our L2/L3 (durable) vs. L0/L1 (volatile)

- PostgreSQL + pgvector as the eventual backend (validates our migration path)

## Finding 4: Contradiction Detection (Open Source, All Components)

**Source:** Perplexity research + Hugging Face model docs

**The pipeline (fully open source):**

```python
# Step 1: Embed all claims
from sentence_transformers import SentenceTransformer
embedder = SentenceTransformer('all-MiniLM-L6-v2')  # Apache 2.0
embeddings = embedder.encode(claims)

# Step 2: Cluster to avoid O(n²)
import hdbscan  # BSD 3-Clause
clusterer = hdbscan.HDBSCAN(min_cluster_size=2)
labels = clusterer.fit_predict(embeddings)

# Step 3: NLI within clusters
from transformers import AutoTokenizer, AutoModelForSequenceClassification
model = AutoModelForSequenceClassification.from_pretrained(
    'cross-encoder/nli-MiniLM2-L6-H768'  # Apache 2.0
)
# Classify pairs as entailment/neutral/contradiction
# contradiction score > 0.7 → flag
```

**Performance:**

| Metric | Value | Source |
|---|---|---|
| Pairwise reduction | 95–99% (via clustering) | Research |
| NLI accuracy | ~85% on contradiction | DeBERTa/MiniLM2 benchmarks |
| NLI latency | 22ms/inference on CPU | MiniLM2 specs |
| Time for 1000 claims | ~22 seconds on CPU | 1000 × 22ms |
| Cost | $0.00 (local inference) | Open-source models |

**Fallback for ambiguous cases (NLI score 0.4–0.7):** Send to Claude/Gemini for final judgment. Expected: ~5% of pairs, so ~5 LLM calls per 1000 claims.

**Cold-start (<50 claims):** Skip clustering, do pairwise directly. 50 claims = 1,225 pairs × 22ms = 27 seconds. Acceptable.

## Finding 5: Corroboration Detection (No External Dependencies)

**Source:** Perplexity research on semantic dedup + Mem0 internals

**The algorithm (our own implementation, uses only open-source embeddings):**

```python
# For each L1 claim, find semantically equivalent claims
similar = store.search_semantic(claim.text, top_k=10)  # cosine via embeddings
equivalent = [s for s in similar if cosine(s, claim) > 0.85]

# Check source independence
for eq in equivalent:
    if eq.source_type != claim.source_type or eq.source_ref != claim.source_ref:
        # Different source → independent corroboration
        independent_count += 1

# Promotion criterion
if independent_count >= 2:
    # Candidate for L1 → L2 promotion
```

**What counts as independent:**
- Different `source_type` (e.g., one from git, one from Slack) → independent
- Same `source_type` but different `source_ref` (e.g., two different git commits) → independent
- Same `source_type` AND same `source_ref` → NOT independent (dedup candidate)

**Cost:** $0.00 — uses embeddings already computed in Phase 1 + metadata comparison.

## Finding 6: Staleness Decay (Our Own Math, No Dependencies)

**Source:** Perplexity research on temporal decay + MemoryLLM/MemoryBank patterns

**No published domain-specific decay rates exist anywhere.** Every system uses custom values. These are our starting defaults, flagged for calibration:

| Domain | Half-Life | Rationale |
|---|---|---|
| `codebase` | 7 days | Code changes daily; git churn data confirms |
| `competitive` | 14 days | Market moves weekly |
| `operational` | 30 days | Processes change monthly |
| `user` | 90 days | People change slowly |
| `product` | 60 days | Product decisions are medium-durable |

**Decay function (exponential, standard half-life):**
```python
import math
from datetime import datetime, timezone

def decay_confidence(claim, now=None):
    now = now or datetime.now(timezone.utc)
    days = (now - claim.created_at).total_seconds() / 86400
    half_life = DOMAIN_HALF_LIVES.get(claim.domain.value, 30)
    return claim.confidence * math.exp(-0.693 * days / half_life)
```

## Finding 7: Compression — Don't Compress Claims

**Source:** Perplexity research on lossy compression

**The research consensus is clear:** "Structural approaches (tiered storage, raw logs as ground truth) are far more reliable than probabilistic compression."

**The core problem:** You make compression decisions before knowing what future queries will need. A summary that looks complete today is missing the exact detail someone needs tomorrow.

**Published failure modes:**
- **Unverifiable Omission Rate (UOR):** Fraction of queries that fail because evidence was lost in summaries
- **Decisive constraints dropped:** "Allergies may be dropped, leaving the agent unable to justify an answer with traceable evidence"
- **Probability distributions collapsed to point estimates:** Nuance destroyed

**Our strategy: never compress claims. Compress projections.**
- L0 claims → archive to cold table after extraction to L1 (queryable, not in active searches)
- L1/L2/L3 claims → keep at full fidelity forever
- Markdown projections → summarize aggressively (they regenerate from store on demand)
- Expired claims (confidence decayed below 0.1) → archive after 90 days

**Cost:** Storage is cheap. At 10,000 claims × ~500 bytes each = 5MB. Negligible.

## Recommended Architecture: 7-Phase Hybrid Pipeline

All components are open source or our own code. Zero proprietary dependencies.

```
PHASE 1: EMBED (sentence-transformers, local)
├── Embed all new/changed claims with all-MiniLM-L6-v2
├── Store embeddings in claims table (or separate embeddings table)
├── Cost: $0.00 (local inference, ~0.5s per 100 claims)
└── Dependencies: sentence-transformers (Apache 2.0)

PHASE 2: DETECT CONTRADICTIONS (NLI model, local)
├── Cluster claims by embedding similarity (HDBSCAN)
├── For each new claim: retrieve top-K neighbors within cluster
├── Run NLI (cross-encoder/nli-MiniLM2-L6-H768) on pairs
├── Score > 0.7 → write Contradiction to store
├── Score 0.4–0.7 → queue for LLM review (~5% of pairs)
├── Cost: ~$0.01 for LLM review of ambiguous cases
└── Dependencies: transformers, hdbscan (Apache 2.0, BSD 3-Clause)

PHASE 3: DETECT CORROBORATION (embeddings + metadata, no LLM)
├── Find semantically equivalent claims (cosine > 0.85)
├── Check source independence via source_type + source_ref
├── Count independent sources per L1 claim
├── ≥2 independent → candidate for L1→L2 promotion
├── Cost: $0.00
└── Dependencies: none (uses Phase 1 embeddings + store metadata)

PHASE 4: PROMOTE (rules for L0→L1, LLM for L1→L2 validation)
├── L0→L1: Rule-based
│   ├── Has source provenance? (source_type + source_ref non-empty)
│   ├── Has timestamp? (valid_from set)
│   ├── Is atomic? (single assertion, not compound)
│   └── All yes → promote, set promoted_at
│
├── L1→L2: LLM validates corroborated claims
│   ├── Prompt: "Given these independent sources, is this claim validated?"
│   ├── Sources provided as evidence
│   ├── LLM confirms with reasoning → promote to L2
│   ├── LLM rejects → stays at L1 with rejection note
│   └── Cost: ~$0.02–0.05 per candidate (Claude Haiku)
│
├── L2→L3: LLM proposes + human approves (initially)
│   ├── LLM drafts promotion justification
│   ├── Human reviews in Markdown projection (or auto-approve after calibration)
│   └── Approved → promote to L3
│
└── Dependencies: Claude API (pay-per-use, swappable)

PHASE 5: DECAY (pure math, no dependencies)
├── Apply domain-specific exponential decay to confidence
├── Source-linked check for file-backed claims
├── Confidence < 0.1 → set valid_until = now (expired)
└── Cost: $0.00

PHASE 6: ARCHIVE (SQL operations, no dependencies)
├── L0 claims extracted to L1 → move to archive table
├── Expired claims older than 90 days → move to archive table
├── Archive is queryable but excluded from active searches
└── Cost: $0.00

PHASE 7: PROJECT (our code, already built)
├── Regenerate Markdown projections from store
├── MEMORY.md, decision-journal.md, experiment-log.md
├── Already implemented in src/projectors/markdown.py
└── Cost: $0.00
```

## Open Questions (remaining)

1. **Embedding model choice:** MiniLM-L6-v2 is fast but 384-dim. all-mpnet-base-v2 is 768-dim and higher quality. Need to benchmark on our claim corpus to see if the quality difference matters for corroboration detection at cosine 0.85.

2. **Compound claim splitting:** "Feature Y caused 12% regression in segment Z on March 15" is three facts. Should L0→L1 split these? Rule-based splitting (regex for numbers, dates, named entities) handles 80% of cases. LLM for the rest. Cost: ~$0.01 per compound claim.

3. **Consolidation frequency:** Run after every perceiver batch (4x/day) or daily? Recommendation: after every batch — claims sit at L0 for at most 6 hours instead of 24.

4. **NLI model hosting:** Local CPU inference (22s for 1000 claims) vs. hosted API. Local is free and private. Hosted is faster but costs money. Recommendation: local for MVP, revisit if latency matters.

5. **HDBSCAN tuning:** `min_cluster_size=2` is the default. May need adjustment as claim count grows. Monitor cluster quality monthly.

## Trade-offs

| Trade-off | Choice | Why |
|---|---|---|
| Full-LLM (Letta-style) vs. hybrid | Hybrid | Our claims are typed with structured fields. LLM is overkill for metadata comparisons and math. Save LLM budget for where judgment matters (promotion validation). |
| Local NLI vs. hosted NLI | Local | Free, private, 22s/1000 claims is acceptable for a background job. |
| Compress claims vs. compress projections | Projections only | Research consensus: lossy compression of canonical data creates unrecoverable information loss. Projections regenerate. |
| Conservative promotion vs. aggressive | Conservative | Wrong L3 promotions are high-cost (they enter MEMORY.md and influence all future sessions). Better to under-promote. |
| Fixed decay rates vs. adaptive | Fixed with calibration flag | No published adaptive rates exist. Start with domain-specific half-lives, tune from real data. |
