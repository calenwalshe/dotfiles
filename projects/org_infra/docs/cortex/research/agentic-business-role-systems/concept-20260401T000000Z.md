## Summary

Multi-agent AI systems for business operations are real, production-deployed, and past the hype stage. The space divides cleanly into **framework layer** (tools to build role-based agent systems) and **product layer** (vertical SaaS companies that have already built them, predominantly in sales). The framework layer has consolidated around three dominant choices — CrewAI, AutoGen, LangGraph — each optimized for different production profiles. The product layer is heavily biased toward revenue-generating functions (SDR, outreach, pipeline) because these have the highest tolerance for errors and the most direct ROI signal. Non-sales verticals (ops, finance, PM) are slower but actively being built, with significantly higher compliance/auditability requirements.

Task capture, assignment, and lifecycle tracking are solved at the pattern level but not commoditized — most teams build this themselves on top of frameworks. No off-the-shelf solution handles the full task lifecycle for a general-purpose business agent team.

## Findings

### 1. Framework Layer — How Roles and Tasks Actually Work

#### CrewAI
- Agents defined by explicit **role**, **goal**, **backstory**, and **tool set** — all in prompts
- **Delegation model:** Manager agent (or sequential pipeline) assigns tasks based on crew goal
- **Handoff format:** Pydantic/JSON structured outputs parsed between agents
- **Memory:** Short-term (in-task), long-term (cross-session), entity memory (named things)
- **Best for:** Content ops, marketing, research — anything where the role metaphor maps cleanly to org structure
- **Production status:** Prototyping to mid-production; used widely in demos, increasingly in real pipelines
- **Weakness:** Rigid abstraction — can't easily override delegation logic or inject custom retries

#### AutoGen (Microsoft, now Magentic-One)
- Agents defined via **conversational configs** — roles emerge from system prompts and function registrations
- **Delegation model:** Group chat dynamics — agents negotiate, vote, or are explicitly designated
- **Handoff format:** Conversation history with function call results embedded
- **Memory:** Conversation thread only — no native long-term memory
- **Best for:** Research, code execution, exploratory workflows where structure is unclear upfront
- **Production status:** Active research/experimentation, less common in pure production deployments
- **Weakness:** Conversational drift in long multi-turn sessions; complex setup curve; non-deterministic routing

#### LangGraph
- Agents are **nodes in a directed graph** with typed state passing between them
- **Delegation model:** Conditional edges (rule-based or LLM-decided) route state between nodes
- **Handoff format:** Typed state objects — fully inspectable, loggable, resumable
- **Memory:** Persistent graph state; supports mid-graph human interrupts
- **Best for:** Mission-critical workflows requiring determinism, auditability, HITL checkpoints
- **Production status:** Enterprise-grade; backed by LangChain Inc; used by large orgs
- **Weakness:** Verbose code, steep learning curve, LangChain ecosystem churn risk

#### Agency Swarm
- Thin abstraction on top of OpenAI Assistants API — roles defined via `Agent` class with tool files
- **Best for:** Quick OpenAI-native builds; teams already deep in the OpenAI ecosystem
- **Production status:** Less common outside OpenAI-native shops

#### OpenAI Agents SDK (new, 2025)
- Python-native, stateless by default, supports handoffs via explicit `transfer_to_<agent>` tool calls
- Simplest possible mental model for getting started
- **Best for:** Zero-to-one validation and board demos; not production-grade for complex flows

#### Framework Selection Matrix

| Use Case | Recommended Framework | Why |
|----------|----------------------|-----|
| Rapid prototype / board demo | CrewAI or OpenAI Agents SDK | Lowest friction, readable code |
| Client-facing production pipeline | CrewAI | Fast iteration, clean API, good enough control |
| Mission-critical / compliance-required | LangGraph | Deterministic, debuggable, resumable |
| Research / novel architectures | AutoGen | Flexible, conversational discovery |
| OpenAI-native shop | Agency Swarm | Tight ecosystem fit |

| **Mindy** | — | AI executive assistant | Meeting scheduling, email management, task routing |

Most of these are seat-based SaaS products presenting a single "AI team member" face — the multi-agent orchestration is internal to the product.

#### Customer Support
- **Intercom Fin** — LLM-powered support with escalation routing
- **Zendesk AI** — triage + resolution agent with human escalation

#### Operations / IT
- Incident management: automated triage, runbook execution
- Supply chain: inventory forecasting agents
- Mostly "agent-assisted" — human reviews outputs, not full autonomy

#### Finance
- Compliance monitoring, fraud detection support, automated report generation
- High auditability requirements — most deployments are HITL, not autonomous
- ERP/accounting system integration is the hard problem (not the AI)

#### Project Management
- Status update automation, meeting summarization, task dependency analysis
- Integration with Jira/Asana/Linear is the primary coupling point
- No dominant product player yet — largely DIY on top of frameworks

**Key observation:** The product layer is 80% sales/revenue. Every other vertical exists but is in earlier stages. The error tolerance and direct ROI in sales makes it the natural beachhead.

### 3. Core Architectural Patterns

#### Task Capture
- Natural language goal → structured output (JSON/Pydantic) via LLM parsing
- Some systems add explicit form/schema intake for higher-stakes tasks
- Caching and idempotency keys used to prevent duplicate task execution

#### Task Assignment
Three dominant patterns:
1. **Role-based routing** (CrewAI): Manager agent reads task, delegates to specialist based on role profile
2. **Conversational routing** (AutoGen): Agents vote or negotiate assignment in group chat
3. **Graph routing** (LangGraph): Deterministic edges or LLM-driven conditional branches with typed state

#### Task Lifecycle
Full lifecycle in mature systems:
1. **Capture** — NL or structured intake
2. **Decompose** — complex goals split into sub-tasks, often dynamically
3. **Assign** — route to specialist agent
4. **Execute** — agent uses tools, calls APIs, writes outputs
5. **Validate** — automated check or HITL review gate
6. **Handoff** — pass result to next agent or return to user
7. **Re-plan** — if validation fails, agent re-plans and retries (up to N attempts)
8. **Archive** — persist state to DB/vector store for future context

#### State Persistence
- In-memory: all frameworks support this; lost on restart
- Short-term: Redis/conversation history for session-scoped context
- Long-term: Vector DB (Pinecone, Weaviate, Qdrant) for semantic retrieval across sessions
- Relational: task status, assignment history, audit logs
- Mature systems use all four layers with different TTLs

#### Production Infrastructure
- Agents deployed as microservices (Docker + Kubernetes)
- API-first integration into CRM/ERP/ticketing systems
- Observability via LangSmith (LangChain), custom traces, or OpenTelemetry
- HITL via web UIs or Slack/Teams integrations for approval gates

### 4. Failure Modes and Open Problems

#### Framework-Specific
- **CrewAI:** Hidden abstractions obscure retries/manual interventions; rigid for non-team workflows
- **AutoGen:** Conversational drift in long sessions; non-deterministic routing; moderate production readiness
- **LangGraph:** Verbose, steep curve, LangChain ecosystem churn risk

#### Universal Failure Modes
| Failure Mode | Description | Mitigation |
|---|---|---|
| **State loss** | Agent loses context across sessions or after restart | Explicit state persistence layer |
| **LLM non-determinism** | Same input produces different handoff decisions | Structured output schemas, deterministic routing where possible |
| **Context window overflow** | Long agent chains exceed context limits | Summarization agents, chunked state |
| **Cost blow-up** | Unbounded loops, retry storms, expensive models per-hop | Token budgets, model tiering, circuit breakers |
| **Conflicting agents** | Two agents reach inconsistent conclusions and deadlock | Designated arbitrator agent or rule-based resolution |
| **Auditability gap** | Hard to explain why an agent made a decision | Structured logs, LangSmith-style tracing |
| **Tool failure cascade** | One API failure cascades through the agent chain | Explicit error recovery nodes, fallback paths |
| **Overengineering** | Multi-agent system for tasks a single well-tuned prompt could handle | Single-agent-first principle; decompose only when you hit the ceiling |

## Trade-offs

| Decision | Option A | Option B | Recommendation |
|----------|----------|----------|----------------|
| Framework | CrewAI (fast, opinionated) | LangGraph (slow, powerful) | Start CrewAI, migrate to LangGraph when you need HITL or compliance |
| Agent autonomy | Fully autonomous (24/7) | Human-in-the-loop at gates | HITL for any decision with real consequences; autonomy for pure data tasks |
| State backend | In-memory | Full persistence stack | Full persistence required for production; in-memory only for prototypes |
| Role granularity | Broad roles (3-5 agents) | Fine-grained roles (10+ agents) | Start broad; fine-grain only where you observe bottlenecks |
| Vertical | Sales (proven ROI) | Other verticals (higher friction) | Sales beachhead is rational; other verticals require more patience and compliance work |

---

## Recommendations

1. **Don't build a custom orchestration layer** until you've validated on CrewAI or LangGraph. The framework layer is mature enough; the infra cost of rolling your own is high.

2. **The product gap is not in sales** — it's in general-purpose internal ops: task intake, assignment, lifecycle tracking across a business with mixed roles. This is underbuilt. Most products are single-vertical.

3. **Task capture is the unsolved hard problem** for general business agents, not task execution. Getting structured, actionable tasks from messy human input (Slack messages, emails, voice) into an agent queue without losing fidelity is where most production systems break.

4. **If integrating with your existing stack** (GSD, openclaw, Taskwarrior): LangGraph is the right framework — its typed state machine maps directly to the state-machine model GSD already uses. CrewAI would require adapters.

5. **Start with one role, not many.** Every successful multi-agent business system in production started as a single-agent system that grew. "Let's build a full AI company" as a first step is a failure mode.
