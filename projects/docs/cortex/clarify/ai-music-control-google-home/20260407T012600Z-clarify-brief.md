## Idea (verbatim)

i have google network at home i can play music through interfaces here like streaming via that... but I actually wanna control this trough ai and have my collelction played throgh natural text via claude code interface... can you help me think thirough this and build.

## Goal

Control music playback on Google Home/Cast devices throughout the house using natural language commands issued from the Claude Code CLI, playing from the user's own music collection.

## Non-Goals

- Building a standalone music player application or GUI
- Replacing Google Home's native voice assistant for non-music tasks
- Supporting non-Google cast targets (AirPlay, Bluetooth, Sonos native)
- Building a music recommendation engine or playlist generation AI
- Handling music acquisition, ripping, or library management
- Multi-user access control or authentication

## Constraints

- Must work from Claude Code CLI (skill or natural language in conversation)
- Must target Google Cast protocol (the user's existing hardware)
- Must run on the user's local machine (`/home/agent/`) — no external server deployments
- Audio serving must be reachable from the LAN (Cast devices pull audio via HTTP)
- No cloud dependencies beyond what Google Cast itself requires
- Must handle the user's own music collection (not just streaming service delegation)

## Assumptions

- User has one or more Google Home / Nest / Chromecast devices on the local network
- User has a music collection accessible from this machine (local files — format TBD)
- Google Cast protocol is usable from a local Node.js or Python client (e.g., `pychromecast`)
- The machine can serve audio files over HTTP on the LAN
- Claude Code skills can invoke local services to trigger playback

## Open Questions

1. **Where does the music collection live?** Local files on this machine (MP3/FLAC/etc.)? A streaming service library (Spotify, YouTube Music)? Both?
2. **What Google Cast hardware is available?** Google Home Minis, Nest Hub, Chromecast Audio, Chromecast on TVs? Which rooms / device names?
3. **What's the preferred control surface?** A `/play` skill command? Natural language mid-conversation ("put on some jazz in the kitchen")? Both?
4. **Does the collection have good metadata?** ID3 tags, folder structure, or a catalog DB? This affects how natural language queries resolve to tracks.
5. **Multi-room expectations?** Play same thing everywhere, or target specific rooms by name?
6. **Queue / playlist behavior?** "Play this album" should queue tracks? Skip/pause/volume commands needed?
7. **Is there an existing media server running?** (Plex, Jellyfin, Navidrome, etc.) — could shortcut the audio serving layer.

## Next Research Steps

1. Audit the local network for discoverable Google Cast devices (mDNS/SSDP scan)
2. Survey the music collection — location, size, formats, metadata quality
3. Evaluate `pychromecast` library capabilities and limitations for programmatic cast control
4. Research local media server options (lightweight HTTP server vs. full media server) for serving audio to Cast devices
5. Prototype a minimal cast-and-play: pick a file, serve it, cast it to a device — validate the core loop works
6. Design the Claude Code skill interface — command parsing, device targeting, search/match logic
