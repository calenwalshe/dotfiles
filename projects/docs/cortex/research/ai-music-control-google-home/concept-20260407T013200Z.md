## Summary

Controlling Google Cast devices programmatically from Python/CLI is a solved problem with mature tooling. The critical architectural challenge is that **this development machine is a cloud VPS (Vultr, 144.202.81.218)**, not on the user's home LAN. Google Cast is a LAN-only protocol — devices discover via mDNS and pull media via local HTTP. A network bridge between the VPS and home LAN is required before any cast control is possible.

## Findings

### 1. Google Cast Protocol — How It Works

Google Cast devices are essentially headless Chrome browsers. The control flow is:

1. **Discovery**: Devices announce via mDNS (`_googlecast._tcp.local`) on the LAN
2. **Connection**: Client connects to device on port 8009 (TLS, proprietary protobuf protocol)
3. **Media playback**: Client tells device to **fetch a URL via HTTP** — the device pulls the media, it is not pushed to it
4. **Transport control**: Play, pause, seek, volume, queue — all via the Cast protocol channel

**Implication**: Both the controller and the media source must be reachable from the Cast device's LAN. A Cast device will not accept media from a public URL it can't resolve or reach (it uses Google DNS 8.8.8.8, so public URLs work, but the controller must be on the same subnet for discovery and command).

### 2. pychromecast — The Gold Standard Library

- **Repo**: `home-assistant-libs/pychromecast` (maintained by Home Assistant team)
- **Version**: 14.0.5 (Oct 2024), Python 3.11+
- **Status**: Not installed on this machine
- **Capabilities**:
  - Auto-discovery of all Cast devices on LAN (mDNS/Zeroconf)
  - Start Default Media Receiver and play any HTTP-accessible media
  - Full transport controls: play, pause, stop, seek, volume, mute
  - Queue management (add to queue, next/previous)
  - Multi-room / audio group support
  - Custom namespace handlers (extensible)
  - Callback-based and blocking APIs
- **Limitations**:
  - Must be on the same subnet as Cast devices
  - No built-in HTTP server for local files (you supply the URL)
  - Async discovery can be tricky (callbacks, timeouts)

### 3. catt (Cast All The Things) — Best CLI Tool

- **Repo**: `skorokithakis/catt`
- **What it does**: Wraps pychromecast into a clean CLI with automatic local file serving
- **Key features**:
  - `catt cast ./file.mp3` — auto-starts HTTP server (ports 45000-47000), sends URL to Cast device
  - `catt scan` — discover all Cast devices
  - `catt -d "Kitchen" cast ./song.mp3` — target specific device by name
  - `catt pause`, `catt play`, `catt stop`, `catt volume 50` — transport controls
  - `catt.cfg` — persistent device aliases and defaults
  - Supports yt-dlp for streaming URLs
- **Limitation**: Same LAN requirement. Must run on the home network.

### 4. mkchromecast — Alternative (Less Recommended)

- Streams system audio (PulseAudio/ALSA) to Cast devices
- Supports encoding backends: node.js, ffmpeg, parec
- Known compatibility issues with newer pychromecast versions (import errors)
- Better for "stream what's playing on my PC" than "play this specific file"
- Less maintained, more fragile

### 5. Supported Audio Formats (Google Cast)

All Cast devices (Home, Nest, Chromecast Audio) natively support:

| Format | Max Quality |
|--------|-------------|
| FLAC | 96 kHz / 24-bit |
| MP3 | Standard |
| HE-AAC | Standard |
| LC-AAC | Standard |
| Vorbis (OGG) | Standard |
| WAV (LPCM) | Standard |
| Opus | Standard |
| WebM | Standard |

**No transcoding needed** for any common music format. This is a non-issue.

### 6. Home Assistant as a Bridge

Home Assistant's Cast integration + REST API provides a remote-controllable bridge:

- **REST API**: `POST /api/services/media_player/play_media` with entity_id and media URL
- **Entities**: Each Cast device becomes `media_player.<name>` automatically
- **Transport**: play, pause, stop, next, previous, volume, seek — all via REST
- **Media browsing**: Supports Plex, Spotify, local media, arbitrary URLs
- **Requirement**: HA must run on the home LAN. REST API can be exposed via HA Cloud or port forwarding.

### 7. Network Gap — The Architectural Blocker

**Current state of this machine**:
- IP: 144.202.81.218 (Vultr VPS, public internet)
- No VPN/tunnel: No Tailscale, WireGuard, or ZeroTier installed
- No Home Assistant running
- No music files (1 test MP3 found)
- No mDNS/avahi available
- No Cast devices discoverable

**This means**: Direct pychromecast/catt usage from this machine is impossible without a network bridge.

## Trade-offs

### Architecture Option A: Home LAN Agent (Recommended)

Deploy a lightweight service on a home network device (RPi, NAS, spare PC, or even a Docker container on an always-on machine).

| Component | Role |
|-----------|------|
| `music-agent` (Python) | Runs on home LAN. Indexes music, serves HTTP, controls Cast via pychromecast |
| Claude Code skill | Sends commands to music-agent via SSH tunnel or Tailscale |
| Music collection | Stored on home machine, indexed with metadata |

**Pros**: Full control, no third-party dependencies, works offline
**Cons**: Requires a device on the home LAN, SSH/VPN setup

### Architecture Option B: Home Assistant Bridge

Run Home Assistant on the home LAN, use its REST API as the bridge.

| Component | Role |
|-----------|------|
| Home Assistant | Runs on home LAN. Cast integration auto-discovers devices |
| Claude Code skill | Sends REST API calls to HA (via HA Cloud or Tailscale) |
| Music collection | Served via HA media directories or Plex/Navidrome |

**Pros**: Mature, well-supported, handles device management, expandable to other smart home control
**Cons**: Heavy dependency (full HA install), learning curve, overkill if only used for music

### Architecture Option C: Tailscale Mesh + Direct Control

Connect VPS to home LAN via Tailscale, run pychromecast directly from VPS.

| Component | Role |
|-----------|------|
| Tailscale | Mesh VPN connecting VPS to home LAN |
| pychromecast | Runs on VPS, discovers and controls Cast devices via Tailscale |
| HTTP server | Serves music files (must be on LAN or Tailscale-accessible) |

**Pros**: Simplest code, no intermediary service
**Cons**: mDNS over Tailscale is unreliable (Cast discovery may fail), music files must still be LAN-accessible, latency

### Architecture Option D: Run Claude Code Locally

Run Claude Code on a home machine instead of / in addition to the VPS.

**Pros**: Eliminates the network gap entirely, simplest architecture
**Cons**: Changes the operator model, may not be desirable

## Recommendations

1. **Option A is the strongest path** — a lightweight Python agent on the home LAN that Claude Code controls remotely. It's the only option that gives full Cast control without heavy dependencies.

2. **The music collection question is still open** — where the files are determines whether we need file sync, NAS integration, or streaming service APIs.

3. **Prototype in two stages**:
   - Stage 1: Get catt/pychromecast working on a home LAN device, validate cast-and-play with local files
   - Stage 2: Build the Claude Code skill that talks to the home agent

4. **catt is the fastest path to a working prototype** — it handles HTTP serving, device discovery, and transport controls in a single CLI tool.
