## Summary

This project is architecturally feasible with existing, mature tooling. The system has four audio source paths (vinyl, local files, Spotify, YouTube/YT Music) converging through a Raspberry Pi home agent to Google Cast speakers, controlled via Claude Code on a remote VPS over Tailscale. Each source path has different integration mechanics — some trivial, some with sharp edges. The biggest risk isn't technical but operational: Spotify's programmatic Cast control requires workarounds because their API doesn't natively bridge to Cast devices.

## Prior Research (Carried Forward)

The initial dossier (`ai-music-control-google-home/concept-20260407T013200Z.md`) established:
- pychromecast (v14.0.5) is the gold-standard Python Cast control library
- catt wraps pychromecast into a clean CLI with auto HTTP serving
- Google Cast supports MP3, FLAC, AAC, OGG, WAV, Opus natively — no transcoding needed
- This machine is a Vultr VPS — requires a network bridge to the home LAN
- Cast is LAN-only: devices discover via mDNS, pull media via HTTP

## New Findings

### 1. Spotify → Cast: Two Approaches, Both Have Friction

**Approach A: pychromecast SpotifyController**
- pychromecast has a built-in `SpotifyController` class (`controllers/spotify.py`)
- Workflow: authenticate via spotipy (OAuth), get access token, launch Spotify app on Cast device via `SpotifyController`, then use spotipy's `start_playback(device_id=...)` to play
- **Friction**: The Cast device must appear in Spotify's device list, which requires the Spotify app to be "launched" on the Cast device first. This launch sometimes hangs or times out (reported in pychromecast issue #769, Feb 2025). The `spotify-token` library that the official example depends on has been broken for some time.
- **Requires**: Spotify Premium, spotipy, OAuth app credentials (Spotify Developer Dashboard)

**Approach B: Spotify Web API + transfer_playback**
- Use spotipy to search tracks and call `start_playback(device_id=cast_device_id, uris=[...])` or `transfer_playback(device_id=cast_device_id)`
- The Cast device must already be visible to Spotify as a Connect device
- **Friction**: Same chicken-and-egg problem — Spotify won't see a Cast device until Spotify is actively casting to it from *somewhere*. Community consensus (latenode.com, Reddit) is that there's no clean programmatic bridge. Workarounds include headless browser automation (Selenium) or keeping a Spotify session alive.

**Recommended path**: Use pychromecast's SpotifyController for the initial launch, then spotipy for track selection and playback control. Accept that the first-launch flow may need a manual Spotify cast to "wake up" the device. This is a known limitation of Spotify's ecosystem, not something we can engineer around.

### 2. YouTube / YouTube Music → Cast: Clean Path

**pychromecast YouTubeController**
- Built-in controller at `controllers/youtube.py`
- Supports: `play_video(video_id)`, `add_to_queue(video_id)`, `play_next(video_id)`, `clear_playlist()`
- Uses the YouTube Cast receiver app (native to all Cast devices)
- Works for both YouTube videos and YouTube Music tracks (same video IDs)
- No authentication required for public videos
- **For audio-only from YouTube**: The Cast device plays the full video on audio-only devices (Google Home speakers just play the audio track). On screen devices (Nest Hub), it shows the video.

**catt + yt-dlp alternative**
- `catt cast "youtube-url"` uses yt-dlp internally to resolve and cast
- Can also `catt cast` any URL supported by yt-dlp (1000+ sites)

**YouTube Music specifically**:
- YouTube Music tracks are just YouTube videos with music metadata
- Can be cast via the same YouTubeController using the video ID
- YouTube Music playlists: extract video IDs from playlist, queue them via `add_to_queue()`

### 3. Vinyl Capture → Cast: Proven Pattern

**Hardware: Behringer UCA202 (~$19)**
- USB audio interface with RCA stereo input, USB-powered
- 48 kHz / 16-bit converters
- Class-compliant USB audio — no drivers needed on Linux/Pi
- Plug-and-play with ALSA on Raspberry Pi
- Recommended by multiple sources for exactly this use case (vinyl → Pi capture)
- Alternative: Behringer UCA222 (identical internals, different color)
- Higher-end option: HiFiBerry DAC+ ADC Pro (Pi HAT, 192kHz/24-bit, ~$65)

**Software pipeline: ffmpeg → Icecast → Cast**
1. Phono amp RCA out → Y-splitter → one leg to existing A/B speakers, one leg to UCA202
2. UCA202 USB → Raspberry Pi
3. Pi captures via ALSA: `arecord -D hw:1,0 -f cd` or via ffmpeg
4. ffmpeg encodes to MP3/OGG and pushes to Icecast server (also on the Pi)
5. Icecast serves an HTTP stream URL (e.g., `http://pi-ip:8000/vinyl.mp3`)
6. pychromecast tells Cast speaker group to play that stream URL
7. All Cast speakers in the house play vinyl in sync

**Latency**: ~1-2 seconds (Cast buffering). Acceptable for vinyl listening — you won't be syncing it with the physical speakers in the same room, but other rooms will sound fine.

**Music Assistant alternative**: Music Assistant (open source, Docker) has an in-development "Local Audio-In" provider that does exactly this — captures ALSA input and streams to Cast/AirPlay/Sonos. Written by TorraxDev, works with UCA202. Still in beta as of Oct 2025, but actively developed. Could replace the custom ffmpeg+icecast pipeline.

### 4. Tailscale: Pi as Subnet Router

**Setup**:
1. Install Tailscale on both the VPS and the Pi
2. Configure Pi as a **subnet router** advertising the home LAN (e.g., `192.168.1.0/24`)
3. Enable IP forwarding on the Pi
4. VPS can then reach all home LAN devices via Tailscale

**mDNS over Tailscale**: Tailscale does NOT relay mDNS by default. Cast device discovery via pychromecast won't work directly from the VPS through Tailscale. **Solution**: The Pi runs pychromecast locally and exposes an API. The VPS doesn't need mDNS — it talks to the Pi's API, not directly to Cast devices.

**Architecture implication**: The Pi is the Cast controller, not the VPS. The VPS sends high-level commands ("play Miles Davis in the kitchen"), the Pi resolves them (find track, find device, cast).

### 5. Music Assistant — The "Batteries Included" Option

**What it is**: Open-source music library manager + multi-room streaming server. Runs on Docker (or HA add-on) on Pi or similar.

**What it handles**:
- Connects to Spotify, YouTube Music, local files, Plex, Tidal, etc.
- Streams to Google Cast, AirPlay, Sonos, DLNA, Squeezebox, Snapcast
- Unified library: merges local + cloud tracks, deduplicates
- Multi-room sync via Cast groups
- Local Audio-In provider (in development) for vinyl capture
- REST API for external control
- Has a Home Assistant integration but **does NOT require HA** — runs standalone

**Pros**: Handles 80% of what we're building out of the box. Spotify, YouTube Music, local files, Cast output, multi-room — all built in.
**Cons**: Adds a significant dependency. May not expose enough API surface for the "natural language → specific action" control we need. The LLM control layer still needs to be built on top.

**Verdict**: Worth evaluating seriously. If Music Assistant's API is rich enough, we could build the Claude Code skill as a thin NL→API translation layer on top of MA, instead of building the entire audio pipeline from scratch.

### 6. Claude Code Skill Design

The skill needs to handle these commands naturally:

| User says | System does |
|-----------|-------------|
| "play some Miles Davis in the kitchen" | Search local library → cast to Kitchen speaker |
| "put on my vinyl everywhere" | Start icecast stream → cast to "All Rooms" group |
| "play Coltrane on Spotify in the living room" | SpotifyController → launch on Living Room → spotipy play |
| "play this youtube video" + URL | YouTubeController → play_video on default device |
| "pause" | mc.pause() on active device |
| "turn it up" | cast.set_volume(current + 0.1) |
| "move the music to the bedroom" | transfer playback to Bedroom speaker |
| "what's playing?" | mc.status → format response |

The Pi agent exposes an API with structured endpoints. Claude Code skill translates NL → API call.

## Trade-offs

### Build Custom vs. Use Music Assistant

| Factor | Custom (pychromecast + ffmpeg + icecast) | Music Assistant |
|--------|------------------------------------------|-----------------|
| Control | Full — every line of code is ours | Limited to MA's API surface |
| Complexity | High — we build the audio pipeline | Low — MA handles it |
| Vinyl capture | Manual ffmpeg+icecast setup | Built-in (beta) Audio-In provider |
| Spotify | We wrangle SpotifyController | Built-in provider |
| YouTube Music | We use YouTubeController | Built-in provider |
| LLM integration | We build the API layer | We build NL→MA API translation |
| Maintenance | We maintain everything | MA community maintains the core |
| Learning | Deep understanding of Cast protocol | Black-box for audio internals |

**Recommendation**: Start with Music Assistant. If its API is sufficient for LLM control, we avoid building 70% of the system. If not, we have the knowledge to build custom. The Claude Code skill (NL → commands) is the same work either way.

### Hybrid approach
Use Music Assistant for Spotify, YouTube Music, and local file management. Use custom pychromecast for vinyl streaming (since MA's Audio-In is still beta). Migrate vinyl to MA when their provider stabilizes.

## Recommendations

1. **Set up the Raspberry Pi** with Tailscale + Docker as the home agent
2. **Deploy Music Assistant** in Docker on the Pi — evaluate its API for LLM control
3. **Buy a Behringer UCA202** (~$19) + RCA Y-splitter (~$5) for vinyl capture
4. **Build the Claude Code skill** as NL → Pi API calls over Tailscale
5. **Start with what works out of the box** (Spotify, YouTube Music, local files via MA), add vinyl capture second
6. **The Spotify first-launch problem** is a known ecosystem limitation — document it as a one-time setup step rather than trying to fully automate it

## Open Questions (Remaining)

1. **Music Assistant API surface** — is it rich enough for "play X on Y" commands? Needs hands-on evaluation.
2. **Pi model** — Pi 3B+ can handle this but Pi 4 is more comfortable (Docker + Icecast + Tailscale + MA). Which do you have?
3. **Phono amp outputs** — does it have a spare RCA out (tape out), or do we need a Y-splitter before the amp?
4. **Spotify Developer credentials** — do you have a Spotify app registered? Need client_id + client_secret for spotipy.
5. **Cast speaker inventory** — need device names for skill command routing. Will get this from `catt scan` once Pi is on the network.
