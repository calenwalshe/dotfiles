## Goal

A live-streaming autonomous DJ system on YouTube that plays continuous music over a visual mosaic of worldwide webcam feeds, controllable via chatbot commands to influence track selection, mood, and visual layout in real time.

---

## Non-Goals

- Not a full DAW or music production tool — this mixes/plays existing tracks, not creates new ones
- Not a video editor for post-production — all compositing happens live in real time
- Not a multi-platform simultaneous streamer (YouTube first, others later if at all)
- Not a replacement for human DJing — the human influences via chatbot, the system handles execution
- Not a commercial music licensing platform — must work within legal/free music boundaries

---

## Constraints

- Must comply with YouTube's music copyright policies (Content ID, DMCA) — this is the hardest constraint and drives music source selection
- Must use publicly accessible webcam feeds (no private cameras, no scraping login-walled streams)
- Must run autonomously on existing server infrastructure (home agent stack)
- Chatbot integration must work with existing bot infrastructure (openclaw or similar)
- Stream must maintain stable video output even when individual webcam feeds drop
- YouTube live stream API rate limits and terms of service must be respected

---

## Assumptions

- YouTube allows continuous live streams with music if licensing is handled correctly (e.g., royalty-free, Creative Commons, or platform-licensed music)
- Sufficient public webcam RTSP/HLS streams exist to create an interesting visual experience
- The server has enough CPU/GPU headroom to transcode multiple webcam feeds + audio into a single live stream
- FFmpeg or similar can handle the real-time compositing pipeline
- The chatbot can send control signals (track change, mood shift, layout change) to the streaming pipeline with low enough latency to feel responsive

---

## Open Questions

- What music sources are legally safe for YouTube live streams? (royalty-free libraries, Creative Commons, Mixcloud-style licensing, AI-generated music, own productions?)
- What are the best sources for public webcam feeds? (EarthCam, Windy, DOT cams, YouTube live cams, RTSP directories?)
- What's the right compositing architecture? (FFmpeg filter_complex, OBS headless, GStreamer, custom?)
- How should the chatbot influence music? (genre/mood commands, specific track requests, tempo/energy levels, transition triggers?)
- What visual layouts work? (grid, single-focus-with-thumbnails, crossfade between cams, geographic themes?)
- How much server resource does real-time multi-source compositing + YouTube RTMP push actually require?
- Should the DJ logic be rule-based (playlist + transitions) or ML-driven (mood detection, beat matching)?
- How to handle webcam feed failures gracefully in the live composite?

## Next Research Steps

1. Research YouTube live stream music copyright rules — what's actually allowed, what triggers Content ID, what music sources are safe
2. Survey royalty-free / Creative Commons / AI-generated music sources and APIs suitable for automated DJ systems
3. Catalog public webcam feed sources — protocols (RTSP, HLS, MJPEG), reliability, geographic coverage, API access
4. Evaluate real-time video compositing pipelines — FFmpeg, OBS headless, GStreamer, Liquidsoap — for multi-source live mixing
5. Research existing projects doing similar things (lo-fi streams, 24/7 YouTube radio stations, webcam aggregators) for architecture patterns
6. Assess chatbot-to-pipeline control mechanisms — how to wire chat commands into audio/video pipeline state changes in real time
7. Estimate server resource requirements for the target architecture
