# Research Dossier: self-hosted-radio — Concept

## Summary

A self-hosted internet radio station on the agent-stack box is technically straightforward and
maps cleanly onto two Docker containers (Icecast2 + Liquidsoap) integrated into the existing
Caddy reverse proxy. The main architectural discovery is that Caddy — not Nginx — is the existing
proxy and already owns 80/443, so no new proxy infrastructure is needed. Port 8000 is occupied;
Icecast must use an alternate port (exposed only internally, proxied by Caddy). Webcam feeds
require a lightweight server-side proxy sidecar because Windy's API is CORS-blocked from the
browser. AzuraCast is not worth it for this use case. Bare stack is correct.

---

## Findings

### q1 — Agent-stack box specs [codebase]

- **CPU:** Intel Xeon (Skylake) x86_64, 5 cores — not ARM, no cross-compilation issues
- **RAM:** 16 GB total, ~9 GB available
- **Disk:** 300 GB total, 84 GB free — plenty for MP3 library + service images
- **Docker network:** all services join `agent-stack` external network
- **Existing reverse proxy:** Caddy (v2), owns ports 80 and 443, domain `calenwalshe.com`
- **Port 8000:** OCCUPIED by `magmalab-api` — Icecast needs a different port
- **Redis:** already running on the agent-stack network — available to radio services if needed
- **Existing services on high ports:** 9080–9600 range is dense. Safe free bands: 8100–8199

**Practical implication:** Icecast should bind internally to port 8100 (or similar), exposed only
within the Docker network. Caddy proxies `radio.calenwalshe.com` → internal Icecast. This keeps
port mapping clean and routes HTTPS automatically via Caddy's Let's Encrypt.

### q2 — External from day 1 [factual]

Confirmed by user. Caddy's automatic HTTPS + Let's Encrypt means this is trivial — add a new
`radio.calenwalshe.com` block to Caddyfile and Caddy handles the cert. The Icecast stream itself
(`/stream.mp3`) and the web page are served through Caddy. No manual cert management.

### q3 — Visual aesthetic [factual]

User confirmed: minimal. A dark, single-page design with the audio player (HTML5 `<audio>`
element pointed at the Icecast stream URL), current track metadata, and embedded webcam feed(s).
No framework needed — static HTML/CSS/JS is sufficient.

### q5 — Icecast2 + Liquidsoap Docker Compose pattern [mechanism]

**Standard pattern:**
- `moul/icecast` — well-maintained community image, env-var driven config
- `savonet/liquidsoap:latest` — official upstream image, x86_64-compatible, no PulseAudio needed
- Liquidsoap uses `output.icecast` to push encoded audio over TCP to the Icecast container by
  service name. This is a pure network push — no sound card, no `/dev/snd`, no PulseAudio needed
  for a file-based playlist setup.

**Minimal `radio.liq`:**
```
radio = playlist("/music")
output.icecast(%mp3(bitrate=128),
  host="icecast", port=8100,
  password="SOURCE_PASSWORD",
  mount="/stream.mp3",
  radio)
```

**Gotchas:**
1. Startup race: Liquidsoap starts before Icecast is ready. Add `depends_on` + reconnect
   retry in `.liq` (`output.icecast` reconnects by default, but add a `fallback` track).
2. Empty playlist crash: Liquidsoap dies if the music volume is empty or unreadable at start.
   Add a `single("fallback.ogg")` safety track; mount the music volume before starting.
3. Liquidsoap 2.x syntax: `%mp3` not `%mp3.fxp`; check tutorial era before copying config.
4. Source limit: Icecast defaults to 2 concurrent sources. Increase if using multiple mounts.
5. Do not mount `/dev/snd` or use `--privileged` for a file-only playlist setup.

**Smart queue integration:** The existing `dj_mixer.py` generates a sequenced playlist (BPM/key
matched). Liquidsoap can read a dynamic playlist file that `dj_mixer.py` rewrites on each queue
update. Alternatively, Liquidsoap's `harbor` HTTP server can receive track-skip commands from
a Python controller. Both patterns avoid rebuilding the queue logic.

### q6 — Webcam embed approaches [comparison]

**Primary approach: server-proxied img-tag refresh**

Windy v3 API returns signed JPEG snapshot URLs that expire in 10 min (free tier) / 24h (pro).
These API calls are CORS-blocked from the browser (reported in Windy community). The correct
pattern:

1. Small sidecar service (Python FastAPI, ~30 LOC) exposes `/cam-url/{id}` — calls Windy API
   server-side, returns the signed JPEG URL. Hides the API key from the browser.
2. Frontend: `<img id="cam">` loads the URL. JavaScript calls `/cam-url/{id}` every 8 minutes
   to refresh the signed URL (before the 10-min expiry). `setInterval` at 60s swaps `img.src`
   to force a new JPEG frame (JPEG CDN URLs allow cross-origin `<img>` loads without CORS).
3. `onerror` handler on `<img>`: on 401/404, show placeholder or skip to next cam.

**Windy iframe embed:** Documents a weather map embed, not individual webcam feeds. Not usable.

**HLS.js:** Windy confirmed "live stream is not possible via our API." No HLS exposed. Use for
DOT cams that explicitly offer HLS.

**MJPEG:** Windy doesn't expose MJPEG. Browser support inconsistent. Skip.

**DOT traffic cams:** Plain `<img src="DOT_JPEG_URL" onerror="...">` + 30–60s setInterval.
No proxy needed; DOT JPEG URLs are public and don't expire.

**Implementation:** the existing `webcam_client.py` already fetches from Windy. The new sidecar
is a thin wrapper that calls this logic via HTTP instead of being called directly from Python.
Or: extend `src/web/app.py` (already exists) with a `/cam-url` route.

### Landscape — AzuraCast vs bare stack [landscape]

**Verdict: bare stack is correct.** AzuraCast wraps Icecast + Liquidsoap in a 10-container
stack (~1–2 GB images, 300–500 MB RAM idle) with a web GUI for playlist scheduling, AutoDJ, and
multi-user access. None of those features are needed here. The owner already has a Liquidsoap
config and wants a custom aesthetic — AzuraCast's frontend is fixed-theme. Bare stack is
2 containers, under 100 MB RAM, and the owner controls every detail.

The one AzuraCast feature worth noting: its JSON "Now Playing" API. This can be replicated with
Icecast's built-in `/status-json.xsl` endpoint or Liquidsoap's harbor HTTP server — both are
included in the bare stack with zero additional complexity.

| Frontend | Static HTML | Flask/FastAPI | Static HTML served by Caddy — minimal, correct |

## Recommendations

1. **Icecast on port 8100 internally.** Caddy proxies `radio.calenwalshe.com` → `icecast:8100`
   for both the stream endpoint and the web page. Port never exposed to host.

2. **Liquidsoap 2.x + `output.icecast` TCP push.** No PulseAudio. Startup retry built-in.
   Add a fallback silence/tone track so Liquidsoap never dies on empty playlist.

3. **Reuse `src/web/app.py` for the Windy cam-url proxy route.** Add `/cam-url/{cam_id}` that
   calls Windy v3 API server-side and returns the signed JPEG URL. Frontend does img-tag refresh.

4. **Static HTML frontend.** HTML5 `<audio>` player pointing at `https://radio.calenwalshe.com/stream.mp3`.
   Minimal dark CSS. Current track metadata polled from Icecast's `/status-json.xsl`.
   One webcam embed cycling on a 2–5 minute timer using the cam-url proxy.

5. **Smart queue via dynamic playlist file.** `dj_mixer.py` writes `/queue/playlist.m3u` on each
   queue update. Liquidsoap watches this file with `playlist.reloadable`. No new control protocol.

6. **Set `<public>0</public>` in icecast.xml.** Prevents the stream from being indexed by
   public Icecast/Shoutcast directories. Essential for staying off the public radar.

## Adjacent Findings

**Icecast public directory listing is on by default.** Icecast registers streams with public
directories (icecast.org, shoutcast.com) unless `<public>0</public>` is set in the mount config.
Without this, the stream can be discovered by anyone searching those directories. Matters here
because the intent is "pirate radio" (small private audience) — accidental public indexing
defeats the purpose and increases exposure. Set this in the initial Icecast config.

**Caddy owns 80/443 — no new proxy infrastructure needed.** The clarify brief assumed we'd
need to set up Nginx. Caddy is already running and handles Let's Encrypt automatically. Adding
`radio.calenwalshe.com` is one Caddyfile block. This also means the radio frontend can be a
static directory served directly by Caddy with no Python process for the HTML, and only the
cam-proxy route needs the Flask/FastAPI app.

**Windy free tier: 1000 API calls/day.** At one cam-url refresh every 8 minutes for a single
cam = 180 calls/day — comfortably within the free limit. But: if the frontend rotates through
multiple cams (e.g., 10 cams, rotating every 2 minutes), that's 720 signed-URL fetches/day,
still under the limit. Exceeding 1000/day requires Windy Pro. Start free; add rate-limit logging
to the proxy endpoint from day one so you know when you're approaching the ceiling.
