## Summary

The strongest path for automated DJ mixing is **Mixxx in headless mode** with the community `rdircio/autodj` controller script, which provides beat-matched crossfading, harmonic mixing (Camelot wheel), randomized EQ transitions, and effects — all running headless on Linux. Liquidsoap is not suitable for beat-matched mixing (volume-based crossfades only), but remains useful as the final audio playout/routing layer. Pre-analyzing the track library with **essentia** (BPM, key, energy) and feeding that metadata into the mixing pipeline produces the best results with minimal runtime cost.

## Findings

### Liquidsoap: Radio Playout, Not DJ Mixing

- Crossfading is **volume-level analysis only** (dB thresholds) — no BPM sync, beat detection, or beat-grid alignment
- **Autocue** (v2.2.5+, May 2024): uses FFmpeg to compute optimal cue points, overlap duration, and loudness normalization (EBU R128). Dramatically improves transition quality but remains a loudness heuristic, not musical
- No harmonic mixing, no key detection, no tempo adjustment, no phrase detection
- Playlist scheduling is powerful (time predicates, weighted rotation, fallback chains, external scheduler integration via `request.dynamic.list`)
- **Verdict:** Use for final audio routing to OBS/stream, not for mixing decisions

### Mixxx Headless: The Right Tool

- Supports `--headless` for CLI/server operation on Linux — no GUI required
- Built-in Auto DJ: loads tracks from playlist, crossfades automatically, supports BPM sync during transitions
- Configurable transition types: crossfade, cut, spinback, brake, echo fadeout
- **`rdircio/autodj` community script (March 2026):** Adds beatmatching, harmonic mixing, randomized EQ transitions, random effects, and preferences dialog. The most complete open-source auto-DJ scripting effort
- Control via JavaScript MIDI scripting — hundreds of controls for decks, EQ, effects, crossfader, sync, keylock
- No native REST/Python API — control requires MIDI/OSC bridging or JS scripting
- **Verdict:** Primary mixing engine. Run headless, feed tracks via Auto DJ queue

### Beat Detection & Key Analysis Libraries

| Library | Accuracy (MIREX) | Speed | Key Detection | Notes |
|---------|-----------------|-------|---------------|-------|
| **madmom** | Best | Slow (neural net) | No | State-of-the-art beat/downbeat tracking, less maintained |
| **essentia** | Very good | Fast (C++ core) | Yes (`KeyExtractor`) | Full MIR suite, production-grade, used by Freesound/Yamaha |
| **librosa** | Good | Medium | Partial (chroma only) | Pure Python, well-documented, struggles with complex meters |
| **aubio** | Good | Fastest | No | Lightweight, good for real-time |

- **Recommendation:** essentia for pre-analysis (BPM + key + energy in one pass), madmom as fallback for difficult tracks
- **keyfinder** (`mixxxdj/libkeyfinder`): C++ library used by Mixxx itself for key detection

### Transition Strategies for Mixed-Genre Library (90-130 BPM)

**Within BPM neighborhood (< 5% difference):**
- EQ crossfade (swap bass/mids/highs) — best quality, easy to automate
- Long blend (16-32 bars) — works with compatible tracks

**Between BPM neighborhoods (> 5% difference):**
- Filter sweep (HPF/LPF ramp) — masks BPM differences effectively
- Echo-out on outgoing track + clean intro on incoming
- Hard cut — intentional, works for large jumps (> 10 BPM)

**Sequencing strategy:**
- Group tracks into BPM neighborhoods: 90-100, 100-110, 110-120, 120-130
- Constrain Camelot movement to +/-1 or same-key between adjacent tracks
- Build energy in waves: 3-4 tracks ascending, brief dip, rebuild higher
- Use breakdowns/outros for genre transitions where rhythm is sparse

**Key algorithms:**
- Bittner (ISMIR 2017): Automatic playlist sequencing using beat-aligned features and spectral similarity scoring
- DJ.Studio: Timeline automation with transition shape presets + stem separation
- `JoeDominik/Making-A-Harmonic-DJ-Set`: Camelot-based sequencing algorithm

## Trade-offs

### Option: Mixxx Headless + rdircio/autodj Script
**Pros:** Beat-matched transitions out of the box, harmonic mixing, EQ automation, effects, proven DJ engine, community-maintained. Headless on Linux.
**Cons:** No Python/REST API — control via MIDI/OSC/JS scripting. One more process to manage. Learning curve for Mixxx internals.
**Verdict:** selected — most complete solution with least custom code.

### Option: Custom Python DJ Engine (essentia + pydub)
**Pros:** Full control, Python-native, integrates directly with Redis command bus. No external dependencies beyond audio libraries.
**Cons:** Significant engineering to get DJ-quality transitions. Beat alignment, EQ crossfading, and effect automation are hard to get right from scratch. Reinventing what Mixxx already does.
**Verdict:** rejected — too much custom work for marginal benefit over Mixxx.

### Option: Liquidsoap with Autocue (Enhanced)
**Pros:** Already in the stack, excellent radio-style playout, autocue handles cue points and loudness. Simple.
**Cons:** No beat matching. Transitions are fundamentally time-based, not musical. Cannot do harmonic mixing or EQ crossfades.
**Verdict:** retained as routing layer — Mixxx outputs audio, Liquidsoap routes to stream. Or use Liquidsoap alone as "good enough" fallback when Mixxx is unavailable.

### Option: Pre-generate Mixed Sets Offline
**Pros:** Perfect transitions (can iterate), no real-time constraints, could use Mixxx GUI offline then stream the result.
**Cons:** Not dynamic — can't respond to chatbot commands. Fixed sets get repetitive. Storage-heavy.
**Verdict:** deferred — useful for special occasions but not the autonomous mode.

## Adjacent Findings

- **`rdircio/autodj` Mixxx controller script (March 2026):** A community-contributed Mixxx Auto DJ extension that adds beatmatching, harmonic mixing (Camelot wheel), randomized EQ transitions, and random effects to Mixxx's built-in Auto DJ. This is a drop-in solution that eliminates the need to build a custom mixing engine — the build decision shifts from "write a DJ engine" to "configure Mixxx + this script + pre-analyze tracks." Source: github.com/rdircio/autodj

## Open Questions

- How stable is Mixxx `--headless` for 24/7 operation? Memory leaks? Crash frequency?
- How to bridge Mixxx control to the Redis command bus? MIDI-over-network, OSC, or a JS controller script that listens on a socket?
- Can Mixxx's Auto DJ queue be dynamically reordered based on chatbot mood/genre commands?
- What's the latency from Redis command to audible change when going through Mixxx?
- Should we pre-analyze all 42 tracks with essentia and store BPM/key/energy in a JSON database, or let Mixxx analyze on load?
- How does Mixxx headless handle audio output routing to OBS or the RTMP pipeline?
