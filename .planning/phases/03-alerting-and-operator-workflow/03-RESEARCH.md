## Summary

Phase 3 adds three capabilities on top of the Phase 2 foundation: drift-triggered failure alerts, a bypass health digest with threshold alerting, and a Telegram-based staging queue promotion workflow. All three are implemented as additions to the scheduler codebase — no new containers, no new databases.

The alerting path is a post-run hook in `l2_perception.py`. After the suite's `run()` returns, the runner calls `reporter.report()` via `_ReporterAdapter`. The existing `Reporter` class already sends per-run Telegram digests with consecutive-failure counts and new-failure detection. Phase 3 either extends this existing path (adding bypass health and consecutive-failure context) OR adds a separate alerting pass before returning results from `run()`. Both approaches are valid; extending the existing reporter path is lower-friction.

The bypass health digest is a query against `harness.site_test_scores` for rows in the last 24 hours where `suite = 'l2_perception'`. Success rate = rows where `passed=true AND challenge_detected=false` + rows where `passed=true AND challenge_detected=true` (bypass succeeded). Failed bypass = `challenge_detected=true AND passed=false`. This metric is INFR-02.

The staging queue promotion UX resolves to a Python bridge script (`promote_bridge.py`) in `openclaw-fresh/workspace/tools/`, matching the established `screenshot_agent_bridge.py` pattern exactly. The bridge calls `POST http://openclaw-scheduler:9100/harness/sites` with `{"site_key": ..., "base_url": ..., "active": true}`. The `main` agent in openclaw-fresh invokes it via subprocess exec, same as the screenshot tool. This is the correct path — the harness API at `:9100` already exposes `POST /harness/sites` (upsert_site) and `GET /harness/sites?active_only=false` (list sites). Inline keyboard buttons are NOT the right approach because the `main` agent's command dispatch is exec-based, not webhook-based. A `/list-staging` and `/promote <site_key>` text command pair is the correct pattern.

**Primary recommendation:** Add a `_send_failure_alert()` function to `l2_perception.py` that calls `notifier.send_message()` directly for per-URL drift alerts, add a `_compute_bypass_health()` function that queries site_test_scores for 24h success rate, and add a `promote_bridge.py` tool in openclaw-fresh that calls the scheduler's existing `/harness/sites` API.

### Pattern 1: Direct Telegram Alert from Suite Run

**What:** Call `notifier.send_message()` directly within `l2_perception.run()` for per-URL drift events, rather than waiting for the post-run reporter. This fires immediately when a URL's classification changes from passing to failing.
**When to use:** After each URL is processed — if drift detected AND new_cls is non-PASS AND consecutive failures exceed threshold.

```python
# Source: notifier.py — send_message() signature confirmed
import notifier  # module-level import at top of l2_perception.py

async def _send_failure_alert(
    site_key: str,
    url: str,
    prev_cls: str,
    curr_cls: str,
    consecutive: int,
) -> None:
    """Send immediate Telegram alert when a URL transitions to failing."""
    msg = (
        f"<b>Perception Alert</b>\n"
        f"URL: <code>{url[:80]}</code>\n"
        f"Site: {site_key}\n"
        f"Was: {prev_cls} → Now: <b>{curr_cls}</b>\n"
        f"Consecutive failures: {consecutive}"
    )
    await notifier.send_message(msg, parse_mode="HTML")
```

**Trigger condition:** `drift_event is not None AND drift_event["prev"] == "PASS_CONTENT" AND curr_cls != "PASS_CONTENT"`

**Consecutive failure count:** Do NOT use `storage.get_consecutive_failures()` — that queries `harness.harness_results` by test_name, not site_test_scores. For l2_perception, query `site_test_scores` directly for the most recent N rows for this site_key and count non-passing rows from the top. See Pattern 3 below.

### Pattern 2: Bypass Health Digest

**What:** After the URL iteration loop completes, compute bypass success rate from `site_test_scores` for the last 24 hours. Send a digest Telegram message. If below threshold, send an additional alert.
**When to use:** Once per run, after all URLs are processed, before returning results.

```python
# Source: storage.py get_site_scores() — confirmed signature
async def _compute_bypass_health(pool, threshold: float = 0.5) -> dict:
    """
    Query site_test_scores for the last 24h, compute CF bypass success rate.
    Returns dict with: total_challenged, bypassed, failed_bypass, success_rate
    """
    import asyncpg
    async with pool.acquire() as conn:
        rows = await conn.fetch(
            """
            SELECT challenge_detected, passed
            FROM harness.site_test_scores
            WHERE suite = 'l2_perception'
              AND ts >= NOW() - INTERVAL '24 hours'
            """,
        )

    challenged = [r for r in rows if r["challenge_detected"]]
    total_challenged = len(challenged)
    bypassed = sum(1 for r in challenged if r["passed"])
    failed_bypass = total_challenged - bypassed
    success_rate = (bypassed / total_challenged) if total_challenged > 0 else 1.0

    return {
        "total_challenged": total_challenged,
        "bypassed": bypassed,
        "failed_bypass": failed_bypass,
        "success_rate": success_rate,
        "below_threshold": success_rate < threshold and total_challenged > 0,
    }
```

**Digest format:**
```python
async def _send_bypass_digest(health: dict, run_id: str) -> None:
    rate_pct = int(health["success_rate"] * 100)
    status = "ALERT: below threshold" if health["below_threshold"] else "OK"
    msg = (
        f"<b>CF Bypass Health (24h)</b> [{status}]\n"
        f"Challenges: {health['total_challenged']}\n"
        f"Bypassed: {health['bypassed']} / Failed: {health['failed_bypass']}\n"
        f"Success rate: {rate_pct}%\n"
        f"Run: <code>{run_id}</code>"
    )
    await notifier.send_message(msg, parse_mode="HTML")
```

### Pattern 3: Consecutive Failure Count for site_test_scores

**What:** Count consecutive non-PASS classifications for a site_key from `site_test_scores`, newest first.
**Why different from `get_consecutive_failures()`:** That method queries `harness.harness_results` by suite+test_name. For l2_perception, the per-URL pass/fail is in `site_test_scores` under `classification` and `passed` columns.

```python
async def _get_consecutive_failures_for_site(pool, site_key: str) -> int:
    """Count consecutive non-PASS rows for site_key in site_test_scores."""
    async with pool.acquire() as conn:
        rows = await conn.fetch(
            """
            SELECT passed
            FROM harness.site_test_scores
            WHERE site_key = $1
              AND suite = 'l2_perception'
            ORDER BY ts DESC
            LIMIT 20
            """,
            site_key,
        )
    count = 0
    for row in rows:
        if not row["passed"]:
            count += 1
        else:
            break
    return count
```

### Pattern 4: promote_bridge.py — Staging Queue Promotion Tool

**What:** Python script in `openclaw-fresh/workspace/tools/` that the `main` agent calls via exec when the operator types `/promote <site_key>`. Calls `POST http://openclaw-scheduler:9100/harness/sites` to flip `active=True`.

**Why this approach:** The `main` agent in openclaw-fresh has `security: "full"` and exec-dispatches workspace tools as Python subprocesses. This is proven by `screenshot_agent_bridge.py`. The scheduler HTTP API already exposes `/harness/sites` (POST upsert) and `/harness/sites?active_only=false` (GET list). No new endpoints needed.

```python
#!/usr/bin/env python3
"""promote_bridge.py — List or promote staging URLs in the perception harness.

Usage:
    python3 promote_bridge.py list
    python3 promote_bridge.py promote <site_key>

Output: JSON result to stdout for agent consumption.
"""

import json
import sys
from urllib import request as urllib_request, error as urllib_error

SCHEDULER_URL = "http://openclaw-scheduler:9100"


def list_staging() -> dict:
    """GET /harness/sites?active_only=false — return all sites including staging."""
    req = urllib_request.Request(f"{SCHEDULER_URL}/harness/sites?active_only=false")
    try:
        with urllib_request.urlopen(req, timeout=10) as resp:
            data = json.loads(resp.read())
        # Filter to staging (active=False) entries
        all_sites = data.get("sites", [])
        staging = [s for s in all_sites if not s.get("active", True)]
        return {"ok": True, "staging": staging, "count": len(staging)}
    except Exception as e:
        return {"ok": False, "error": str(e)}


def promote_site(site_key: str) -> dict:
    """POST /harness/sites with active=True to promote a staging entry."""
    # First, fetch the site's current record to preserve base_url and category
    list_result = list_staging()
    if not list_result.get("ok"):
        return list_result
    staging = {s["site_key"]: s for s in list_result.get("staging", [])}
    if site_key not in staging:
        # Try all sites (might already be active)
        return {"ok": False, "error": f"Site '{site_key}' not found in staging queue"}

    site = staging[site_key]
    body = json.dumps({
        "site_key": site_key,
        "base_url": site["base_url"],
        "category": site.get("category", "general"),
        "active": True,
        "priority": site.get("priority", 100),
        "notes": site.get("notes"),
    }).encode()

    req = urllib_request.Request(
        f"{SCHEDULER_URL}/harness/sites",
        data=body,
        headers={"Content-Type": "application/json"},
        method="POST",
    )
    try:
        with urllib_request.urlopen(req, timeout=10) as resp:
            result = json.loads(resp.read())
        return {"ok": True, "promoted": site_key, "base_url": site["base_url"]}
    except Exception as e:
        return {"ok": False, "error": str(e)}


def main():
    if len(sys.argv) < 2:
        print(json.dumps({"ok": False, "error": "Usage: promote_bridge.py list|promote <site_key>"}))
        sys.exit(0)

    cmd = sys.argv[1].lower()
    if cmd == "list":
        print(json.dumps(list_staging()))
    elif cmd == "promote" and len(sys.argv) >= 3:
        print(json.dumps(promote_site(sys.argv[2])))
    else:
        print(json.dumps({"ok": False, "error": f"Unknown command: {cmd}"}))


if __name__ == "__main__":
    main()
```

### Anti-Patterns to Avoid

- **Using `storage.get_consecutive_failures()`  for site alerts:** That method queries `harness.harness_results` by `suite + test_name`. For l2_perception, consecutive failures are in `site_test_scores` by `site_key`. Use the raw pool query in Pattern 3.
- **Sending an alert on every run for every non-passing URL:** Alert only when classification transitions from passing to failing (`drift_event is not None AND drift_event["prev"] == "PASS_CONTENT"`). Alerting on already-known failures creates noise.
- **Calling the scheduler's `/trigger/l2_perception` endpoint from the promote bridge:** Not needed. The promote operation is synchronous (single upsert) and the response confirms success. No need to trigger a run.
- **Implementing Telegram inline keyboard for promotion:** The `main` agent dispatches tools via exec subprocess, not via Telegram callback webhooks. Keyboard buttons require a separate update handler that does not exist in the current architecture. Use text command dispatch instead.
- **Adding bypass health as a new scheduler job:** The bypass health digest should fire within the l2_perception run itself (after the URL loop), not as a separate scheduled job. It shares the same DB connection and run context.
- **Querying `harness.harness_results` for bypass health:** The bypass-specific fields (`challenge_detected`, `passed`) are in `site_test_scores`, not `harness_results`. `harness_results` records the runner-level pass/fail, not the per-URL perception outcome.

## Common Pitfalls

### Pitfall 1: Alerting on Already-Known Failures

**What goes wrong:** Every l2_perception run (hourly) sends a Telegram alert for every non-passing URL. Operator receives 10+ alerts per hour for the same known failures.
**Why it happens:** Alerting on `cls != "PASS_CONTENT"` without checking whether this is a *new* failure.
**How to avoid:** Alert only when `drift_event is not None AND drift_event["prev"] == "PASS_CONTENT" AND curr_cls != "PASS_CONTENT"`. This fires only on the first run after a URL transitions from passing to failing. Subsequent runs for the same failing URL produce no alert.
**Warning signs:** Telegram flood of repeated alerts with the same site_key after each hourly run.

### Pitfall 2: Missing Pool Reference in Alert Functions

**What goes wrong:** `_compute_bypass_health()` and `_get_consecutive_failures_for_site()` need the asyncpg pool. If called before pool creation or after pool close in `run()`, they raise connection errors.
**Why it happens:** The pool lifecycle is managed inside `run()` with a try/finally. Alert functions must receive the pool as an argument.
**How to avoid:** Pass `pool` as a parameter to all alert-related functions. Never use module-level pool references.
**Warning signs:** `asyncpg.exceptions.TooManyConnectionsError` or `AttributeError: 'NoneType' has no attribute 'acquire'`.

### Pitfall 3: Bypass Health Threshold on Sparse Data

**What goes wrong:** A bypass health alert fires during initial deployment when only 1-2 URLs have been challenged. A single bypass failure = 0% rate, triggering alert even though the system is functioning correctly.
**Why it happens:** Arithmetic on a tiny sample produces misleading rates.
**How to avoid:** Only send a threshold alert when `total_challenged >= min_sample` (recommend 3). If `total_challenged == 0`, success_rate is 1.0 (no alert). If `total_challenged < 3`, send the digest but suppress the threshold-breach alert.
**Warning signs:** Repeated "bypass health ALERT" messages immediately after deployment with total_challenged = 1.

### Pitfall 4: promote_bridge.py Cannot Reach openclaw-scheduler

**What goes wrong:** Bridge calls `http://openclaw-scheduler:9100/harness/sites` but the hostname `openclaw-scheduler` is not resolvable from within the openclaw-fresh container.
**Why it happens:** Both containers are in the `agent-stack` Docker network, but the service name may differ from the container name. Must verify with `docker inspect`.
**How to avoid:** Verify the hostname before coding: `docker exec openclaw-fresh curl -s http://openclaw-scheduler:9100/health`. If that fails, check `docker inspect openclaw-fresh --format '{{.NetworkSettings.Networks}}'` to confirm which network it's on and what hostname `openclaw-scheduler` resolves to.
**Warning signs:** Bridge returns `{"ok": false, "error": "Connection refused"}` or `"Name or service not known"`.

### Pitfall 5: Alert Storm on First Run After Long Downtime

**What goes wrong:** If l2_perception hasn't run for several hours and 10 URLs have been failing, the first run back detects drift for all 10 and sends 10 alerts simultaneously.
**Why it happens:** Drift detection compares current classification against the most recent `site_test_scores` row — if that row is hours old, every URL that was passing before downtime looks like a new failure.
**How to avoid:** No special handling needed — this is correct behavior. A long outage genuinely means the operator hasn't seen these failures. The alert storm is acceptable. If the operator wants to suppress, they can look at timestamps and acknowledge.

### Pitfall 6: HTML Injection in Alert Messages

**What goes wrong:** A URL containing `<`, `>`, or `&` characters causes malformed HTML in the Telegram message, which Telegram rejects or renders incorrectly.
**Why it happens:** `notifier.send_message()` sends with `parse_mode="HTML"`. Special HTML chars in URLs/titles break the message.
**How to avoid:** Wrap variable-content fields in `<code>...</code>` tags (Telegram renders code blocks safely) OR escape with `html.escape()`. For URLs: wrap in `<code>` tags, not raw text. For site_key (controlled alphanumeric key): no escaping needed.

1. **Is `openclaw-scheduler:9100` reachable from openclaw-fresh container?**

2. **Should bypass health digest fire on EVERY run or only when below threshold?**
   - What we know: Success criteria says "a rate below threshold triggers an alert without requiring manual inspection." It does not say whether a passing digest is always sent.
   - What's unclear: Whether the operator wants a daily/per-run confirmation that bypass is healthy, or only alert-on-failure.
   - Recommendation: Send a brief digest every run (cheap, informative) AND send an explicit ALERT when below threshold. The digest confirms the system is working; the alert fires the action path.
